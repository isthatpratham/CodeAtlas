# 1. Purpose

This document serves as the master implementation guide for anyone building CodeAtlas.

Its purpose is to ensure every feature is developed consistently, follows the project vision, and maintains the same engineering standards throughout the codebase.

This document should be referenced before implementing any new feature.

---

# 2. Guiding Principles

Every implementation should follow these principles:

* Build for production from the beginning.
* Prefer simple solutions over complex abstractions.
* Keep code modular.
* Prioritize readability.
* Avoid duplicate logic.
* Reuse existing components whenever possible.
* Build for maintainability rather than speed of implementation.

If two approaches solve the same problem, choose the one that is easier to understand and maintain.

---

# 3. AI Implementation Rules

These rules are specifically intended for AI-assisted development.

### Never invent features

Only implement functionality defined in the documentation.

---

### Never change the architecture

Folder structure, module boundaries, and data flow must remain consistent with the architecture documents.

---

### Respect component reuse

Before creating a new component, verify that an existing component cannot be extended.

---

### Do not introduce unnecessary dependencies

New libraries should only be added when they solve a problem that cannot reasonably be solved with the existing stack.

---

### Preserve consistency

Follow the established:

* Naming conventions
* Design system
* Motion system
* API contracts
* Folder structure

---

### Ask before making architectural changes

Major architectural decisions should not be made during implementation without updating the documentation first.

# 4. Development Standards

Every feature should follow the same lifecycle.

### Step 1

Understand the requirement.

↓

### Step 2

Identify affected modules.

↓

### Step 3

Implement the feature.

↓

### Step 4

Handle loading and error states.

↓

### Step 5

Test functionality.

↓

### Step 6

Review for consistency.

↓

### Step 7

Merge after validation.

No feature is considered complete until it passes through every step.

---

# 5. Code Quality Standards

The codebase should maintain the following standards:

* Strict TypeScript.
* Small, focused functions.
* Reusable utilities.
* Clear naming.
* Predictable file organization.
* No dead code.
* No commented-out legacy code.
* No duplicated business logic.

Every exported module should have a single responsibility.

---

# 6. UI & UX Rules

Every user interaction should provide immediate and meaningful feedback.

General rules:

* Animations should support usability.
* Avoid visual clutter.
* Keep navigation predictable.
* Preserve spatial context during camera movement.
* Do not overwhelm users with excessive information.
* Prioritize readability over density.

The repository map should always remain the primary focus of the interface.

---

# 7. Performance Rules

Performance should be considered throughout development.

Guidelines:

* Avoid unnecessary re-renders.
* Memoize expensive calculations where appropriate.
* Lazy load non-essential resources.
* Keep animations smooth.
* Process heavy computations asynchronously.
* Optimize rendering for large repositories.

User interactions should remain responsive even as repository size increases.

---

# 8. Repository Support Guidelines

Version 1 targets public GitHub repositories.

The implementation should:

* Validate repository URLs.
* Gracefully handle unsupported repositories.
* Tolerate partially supported file types.
* Continue analysis whenever possible instead of failing completely.

Repository analysis should never execute user code.

Only static analysis is permitted.

---

# 9. Definition of Done

A feature is considered complete when:

* Requirements are fully implemented.
* The code follows project conventions.
* Loading states are present.
* Error states are handled.
* Accessibility considerations are addressed.
* No unnecessary dependencies were introduced.
* Documentation remains accurate.
* The implementation has been reviewed.

Completeness is measured by quality, not by the amount of code written.

---

# 10. Future Expansion Rules

CodeAtlas is expected to grow over time.

Future features should:

* Extend existing modules rather than replace them.
* Maintain backward compatibility where practical.
* Reuse existing data models.
* Preserve API consistency.
* Continue following the established design system.

Planned future capabilities include:

* AI-assisted architecture explanations.
* Complexity visualization.
* Repository evolution timelines.
* Contributor ownership maps.
* Private repository support.
* Multi-repository comparison.
* Plugin ecosystem.
* IDE integrations.

Every new capability should feel like a natural extension of CodeAtlas rather than a separate product.

# Final Statement

These ten documents collectively define the foundation of CodeAtlas.

They establish the product vision, user experience, engineering architecture, implementation strategy, technical standards, and development workflow.

All future development should reference these documents to ensure the project remains consistent, maintainable, and aligned with its original vision.
