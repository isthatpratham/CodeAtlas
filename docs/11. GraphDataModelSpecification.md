# 1. Purpose

This document defines the **single source of truth** for every piece of repository data inside CodeAtlas.

Every subsystem uses this model.

* GitHub Parser
* Repository Analyzer
* Graph Generator
* Backend API
* Frontend
* Search
* Inspector
* Minimap
* Camera
* Future AI Features

No component should invent its own structure.

---

# 2. Design Principles

The graph must be:

* Deterministic
* Serializable
* Extensible
* Versioned
* Language Agnostic
* Independent from the renderer

The frontend should never care **how** the graph was generated.

It only renders it.

---

# 3. Graph Architecture

```text
Repository
│
├── Metadata
├── Statistics
├── Layout
├── Nodes
├── Edges
└── Analysis
```

Everything revolves around a single `RepositoryGraph` object.

---

# 4. Repository Model

```ts
interface RepositoryGraph {
  version: string;

  repository: Repository;

  statistics: RepositoryStatistics;

  layout: LayoutMetadata;

  nodes: GraphNode[];

  edges: GraphEdge[];

  analysis: AnalysisMetadata;
}
```

---

```ts
interface Repository {
```

# 5. Node Model

Every object visible on the map is a node.

---

## Base Node

```ts
interface GraphNode {

    id: string;

    type: "root" | "folder" | "file";

    name: string;

    path: string;

    parentId?: string;

    children: string[];

    metadata: NodeMetadata;

    layout: NodeLayout;
}
```

---

## Node Metadata

```ts
interface NodeMetadata {

    extension?: string;

    language?: string;

    size: number;

    depth: number;

    imports: number;

    importedBy: number;

    isEntryPoint: boolean;

    isExternal: boolean;
}
```

---

## Node Layout

```ts
interface NodeLayout {

    x: number;

    y: number;

    width: number;

    height: number;

    collapsed: boolean;
}
```

---

# Node Types

## Root

Exactly one.

Represents the repository.

---

## Folder

Contains:

* Child folders
* Files

---

## File

Leaf node.

Contains:

* Extension
* Language
* Metadata

# 6. Edge Model

Relationships between nodes.

```ts
interface GraphEdge {

    id: string;

    source: string;

    target: string;

    type: EdgeType;

    direction: "forward";

    metadata?: EdgeMetadata;
}
```

---

## Edge Types

```ts
type EdgeType =

    | "hierarchy"

    | "import"

    | "export"

    | "dependency";
```

Version 1 uses:

* hierarchy
* import

The others are reserved.

---

## Edge Metadata

```ts
interface EdgeMetadata {

    weight: number;

    label?: string;
}
```

---

# 7. Layout Model

Global layout information.

```ts
interface LayoutMetadata {

    algorithm: string;

    generatedAt: string;

    nodeSpacing: number;

    layerSpacing: number;

    zoomLevel: number;
}
```

This allows layouts to remain reproducible.

# 8. Camera Model

The frontend camera.

```ts
interface CameraState {

    y: number;

    zoom: number;

    selectedNode?: string;
}
```

Stored only on the frontend.

Never returned by the backend.

Stores information about the analysis process.

```ts
interface AnalysisMetadata {

    analyzedAt: string;

    duration: number;

    parserVersion: string;

    supportedLanguages: string[];
```

---

# 12. Complete API Response

```ts
interface RepositoryResponse {

    success: boolean;

    data: RepositoryGraph;
}
```

Everything needed by the frontend should be inside `RepositoryGraph`.

The frontend should never need to call multiple endpoints just to render the repository.

# 13. Future Compatibility

The data model is intentionally designed to support future features without breaking existing clients.

Future additions may include:

## AI

```ts
aiSummary

architectureDescription

moduleExplanation
```

---

## Git

```ts
lastCommit

commitHistory

contributors
```

---

## Complexity

```ts
cyclomaticComplexity

codeCoverage

maintainabilityIndex
```

---

## Ownership

```ts
owner

team

reviewers
```

---

## Runtime

```ts
callGraph

executionPath

dynamicImports
```

These fields should be added as optional properties or new interfaces, preserving backward compatibility with Version 1.
