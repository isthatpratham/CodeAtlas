# 1. Purpose

The Repository Analysis & Visualization Engine is the heart of CodeAtlas.

Its responsibility is to transform a GitHub repository into structured graph data that can be visualized as an interactive map.

It **does not render the UI**. It only understands the repository and produces a complete representation of its architecture.

---

# 2. Engine Overview

Every repository follows the same processing lifecycle.

```text
GitHub Repository
        │
        ▼
 Repository Fetch
        │
        ▼
 File Discovery
        │
        ▼
 Structure Analysis
        │
        ▼
 Dependency Analysis
        │
        ▼
 Graph Generation
        │
        ▼
 Layout Calculation
        │
        ▼
 Interactive Map
```

The same repository should always generate the same graph and layout, provided its contents have not changed.

---

# 3. Analysis Pipeline

The engine processes repositories in the following order:

### Step 1

Validate GitHub URL

↓

### Step 2

Fetch repository contents

↓

### Step 3

Read folder hierarchy

↓

### Step 4

Identify supported source files

↓

### Step 5

Extract metadata

↓

### Step 6

Analyze imports and exports

↓

### Step 7

Build dependency graph

↓

### Step 8

Generate visual layout

↓

### Step 9

Return graph data to frontend

---

# 4. Repository Fetching

Version 1 only supports **public GitHub repositories**.

The engine should retrieve:

* Repository name
* Owner
* Description
* Default branch
* Folder structure
* File contents (supported files only)
* Repository statistics

No authentication is required for public repositories unless GitHub rate limits necessitate it.

Repository contents should never be modified.

---

# 5. Repository Structure Analysis

The engine constructs a complete directory tree.

For each folder:

Store:

* Name
* Path
* Parent
* Children
* Depth
* Number of files
* Number of subfolders

For each file:

Store:

* Name
* Extension
* Absolute path
* Parent folder
* File size
* Language
* Entry point status (if applicable)

---

# 6. File Parsing

Initially, only commonly used programming languages will be supported.

Examples:

* TypeScript
* JavaScript
* TSX
* JSX
* JSON
* CSS
* SCSS
* HTML
* Markdown

Unsupported files should still appear on the map but will not participate in dependency analysis.

The parser should never execute code.

It should only inspect the source.

---

# 7. Dependency Analysis

The parser should identify relationships between files.

Examples include:

### Imports

```ts
import Navbar from "./Navbar";
```

↓

Relationship

```
App.tsx
        │
imports ▼
Navbar.tsx
```

---

### Exports

Determine which files expose reusable modules.

---

### Entry Files

Detect common project entry points such as:

* index.ts
* index.tsx
* main.ts
* main.tsx
* app.ts
* app.tsx

These should receive a distinct visual treatment in the map.

---

### External Packages

Dependencies from package managers should be recognized separately from internal project relationships.

---

# 8. Graph Generation

After parsing, the repository is transformed into a graph.

The graph consists of:

## Nodes

Represent:

* Root
* Folder
* File

Each node stores:

* ID
* Type
* Name
* Path
* Parent
* Children
* Metadata

---

## Edges

Represent:

* Folder hierarchy
* Import relationships

Each edge stores:

* Source
* Target
* Relationship type

---

The graph becomes the single source of truth for visualization.

---

# 9. Layout Generation

The graph must be converted into a stable spatial layout.

The layout should prioritize readability over compactness.

General rules:

* Parent folders remain close to children.
* Files should stay inside their folders.
* Deeply nested structures should remain visually organized.
* Edge crossings should be minimized where practical.
* Similar repositories should produce consistent layouts.

The exact layout algorithm is implementation-specific but must remain deterministic.

---

# 10. Visualization Rules

## Root

Displayed as the central origin of the repository.

---

## Folders

Represented as grouped regions containing related files.

Nested folders remain visually nested.

---

## Files

Represented as individual nodes within their parent folder.

---

## Import Relationships

Displayed as directional connections between related files.

Hierarchy and dependency relationships should remain visually distinguishable.

---

## Selection

Selecting a node should emphasize:

* Selected node
* Connected nodes
* Direct relationships

Unrelated content should fade slightly to improve focus.

---

# 11. Interactive Map Behaviour

The visualization should support:

* Smooth zoom
* Smooth pan
* Drag navigation
* Click selection
* Hover previews
* Focus transitions
* Animated camera movement

The map should never feel static.

Interactions should remain fluid even on large repositories.

---

# 12. Repository Statistics

The engine should calculate useful repository insights.

Examples include:

* Total files
* Total folders
* Languages used
* Largest folder
* Deepest folder
* Total dependencies
* Internal imports
* External packages
* Maximum folder depth

Statistics should be calculated during analysis rather than on demand.

---

# 13. Performance Strategy

Large repositories require efficient processing.

The engine should:

* Avoid duplicate parsing
* Cache intermediate results where appropriate
* Parse only supported files
* Minimize repeated dependency calculations
* Keep memory usage predictable

Expensive computations should occur once during analysis.

---

# 14. Limitations

Version 1 intentionally excludes:

* Runtime execution analysis
* Call graph generation
* AI architectural summaries
* Dead code detection
* Git history analysis
* Contributor ownership
* Private repositories
* Live repository synchronization

These features are planned for future versions.

---

# 15. Future Enhancements

The architecture should support future capabilities, including:

* Multi-language parsing expansion
* AI-assisted architecture explanations
* Circular dependency detection
* Complexity heatmaps
* Execution path visualization
* Git blame overlays
* Historical architecture evolution
* Multi-repository comparison

These additions should extend the existing analysis pipeline without requiring major redesign.
