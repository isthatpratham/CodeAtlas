# 1. Purpose

This document defines how the frontend communicates with the backend.

The backend is responsible for understanding repositories.

The frontend is responsible for presenting repositories.

The frontend should never parse repositories directly.

---

# 2. API Philosophy

The API should be:

* Simple
* Predictable
* Versioned
* Fully typed
* Stateless
* Consistent

Every endpoint should return a predictable response structure.

---

# 3. System Overview

```text
             Frontend
                 │
                 ▼
          CodeAtlas API
                 │
                 ▼
     Repository Analysis Engine
                 │
                 ▼
          Graph Generation
                 │
                 ▼
          JSON Response
```

The backend performs all expensive computation before returning data to the client.

---

# 4. Analysis Workflow

Repository analysis follows this lifecycle:

```text
Client
    │
Submit Repository URL
    │
    ▼
Validate Request
    │
    ▼
Fetch Repository
    │
    ▼
Analyze Structure
    │
    ▼
Generate Graph
    │
    ▼
Calculate Statistics
    │
    ▼
Return Repository Data
```

The client only receives processed data.

# 5. API Endpoints

## Health Check

```http
GET /api/v1/health
```

Purpose:

Verify that the API is operational.

---

## Analyze Repository

```http
POST /api/v1/repositories/analyze
```

Request

```json
{
  "repositoryUrl": "https://github.com/facebook/react"
}
```

Purpose

Analyze a repository and return its processed representation.

---

## Repository Result

```http
GET /api/v1/repositories/{repositoryId}
```

Purpose

Retrieve an already analyzed repository.

---

## Repository Statistics

```http
GET /api/v1/repositories/{repositoryId}/statistics
```

Returns:

* File count
* Folder count
* Languages
* Dependency count
* Repository size

---

## Graph Data

```http
GET /api/v1/repositories/{repositoryId}/graph
```

Returns the complete graph used by the frontend renderer.

# 6. Request & Response Models

## Standard Success Response

```json
{
  "success": true,
  "data": {},
  "message": null
}
```

---

## Standard Error Response

```json
{
  "success": false,
  "error": {
    "code": "INVALID_REPOSITORY",
    "message": "Repository could not be analyzed."
  }
}
```

Every endpoint should follow this format.

---

```ts
interface Repository {
```

```ts
interface RepositoryStatistics {
```

---

# Graph Model

```ts
interface Graph {
  nodes: GraphNode[];
  edges: GraphEdge[];
}
```

---

# Node Model

```ts
interface GraphNode {
  id: string;
  type: "root" | "folder" | "file";

```

---

```ts
interface GraphEdge {

```

# 7. Graph Data Model

The graph is the primary data structure consumed by the frontend.

## Node Types

* Root
* Folder
* File

Each node should contain sufficient metadata for rendering without requiring additional API calls.

---

## Edge Types

### Hierarchy

Represents the folder structure.

### Import

Represents relationships between files.

Future edge types may include:

* Export
* Dynamic Import
* Call Graph
* Runtime Dependency

# 8. Error Responses

Errors should use consistent codes.

| Code                  | Meaning                                   |
| --------------------- | ----------------------------------------- |
| INVALID_URL           | Invalid repository URL                    |
| REPOSITORY_NOT_FOUND  | Repository does not exist                 |
| PRIVATE_REPOSITORY    | Repository is inaccessible                |
| RATE_LIMITED          | GitHub rate limit reached                 |
| UNSUPPORTED_LANGUAGE  | Repository contains unsupported languages |
| ANALYSIS_FAILED       | Repository analysis failed                |
| INTERNAL_SERVER_ERROR | Unexpected backend error                  |

The frontend should display user-friendly messages instead of raw error codes.

---

# 9. Caching Strategy

To improve performance:

* Cache completed repository analyses.
* Reuse cached results when the repository has not changed.
* Expire cached entries after a configurable duration.
* Avoid repeating expensive parsing operations.

The caching mechanism should remain transparent to the frontend.

---

# 10. Rate Limiting

The backend should:

* Respect GitHub API limits.
* Prevent abuse of public endpoints.
* Return meaningful responses when limits are exceeded.
* Retry transient failures where appropriate.

---

# 11. API Versioning

All endpoints should be versioned.

Current version:

```text
/api/v1/
```

Breaking changes should be introduced through a new API version rather than modifying existing contracts.

---

# 12. Future API Expansion

The API should be designed to support future capabilities without breaking existing clients.

Potential additions include:

* AI architecture summaries
* Repository history
* Git blame data
* Contributor analytics
* Complexity metrics
* Multi-repository comparison
* Authentication and private repository support

These features should extend the existing API while preserving backward compatibility.
