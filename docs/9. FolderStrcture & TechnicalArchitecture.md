# 1. Purpose

This document defines the physical structure of the CodeAtlas codebase.

A well-organized folder structure improves maintainability, scalability, and onboarding. Every directory should have a single, clearly defined responsibility.

---

# 2. Architecture Philosophy

The project should follow these principles:

* Feature-first organization
* Separation of concerns
* Reusable shared components
* Clear module boundaries
* No circular dependencies
* Predictable imports
* Scalable structure

The architecture should allow new features to be added with minimal impact on existing code.

---

# 3. Monorepo Structure

CodeAtlas will be organized as a monorepo.

```text
codeatlas/
│
├── apps/
│   ├── web/
│   └── api/
│
├── packages/
│   ├── ui/
│   ├── types/
│   ├── utils/
│   └── config/
│
├── docs/
│
├── .github/
│
├── package.json
├── turbo.json (or equivalent)
└── README.md
```

### Responsibilities

**apps/web**

Frontend application.

**apps/api**

Backend API and repository analysis engine.

**packages/ui**

Reusable UI components.

**packages/types**

Shared TypeScript interfaces.

**packages/utils**

Common helper functions.

**packages/config**

Shared configuration and constants.

# 8. Environment Variables

Separate frontend and backend configuration.

### Frontend

```env
NEXT_PUBLIC_API_URL=
NEXT_PUBLIC_APP_NAME=
```

### Backend

```env
PORT=
GITHUB_TOKEN=
CACHE_TTL=
```

Guidelines:

* Never commit secrets.
* Validate required variables at startup.
* Use sensible defaults only where appropriate.

# 10. Import Rules

To keep dependencies clean:

* Components may import from `features`, `services`, `hooks`, `types`, `utils`, and `ui`.
* Features should not import from other unrelated features unless there is a documented dependency.
* Services should contain all API requests.
* Business logic should not live inside UI components.
* Avoid deeply nested relative imports; use configured path aliases where appropriate.

Example:

```ts
import { SearchBar } from "@/components/search/SearchBar";
import { useRepository } from "@/features/repository/useRepository";
```

---

# 11. Scalability Guidelines

When adding a new feature:

1. Create a dedicated feature module.
2. Keep UI, logic, and state together where appropriate.
3. Reuse shared packages before creating new utilities.
4. Avoid modifying unrelated modules.
5. Update documentation if the architecture changes.

Every feature should integrate into the existing structure instead of creating parallel patterns.

# 12. Final Architecture Overview

```text
                    CodeAtlas
                        │
        ┌───────────────┴───────────────┐
        │                               │
   Frontend (Web)                  Backend (API)
        │                               │
        │                               │
 Components                  Repository Analysis
 Features                    GitHub Integration
 State                       Graph Generation
 Services                    Caching
        │                               │
        └───────────────┬───────────────┘
                        │
                 Shared Packages
            (UI • Types • Utils • Config)
```

This architecture provides a clear separation between presentation, business logic, and repository analysis while remaining flexible enough for future features such as AI-assisted explanations, private repository support, and collaboration tools.
